Creating the Phoenix App

Here's how to generate the scaffold Phoenix 1.3 app we walked through in the video:

  1. Start by installing Hex (Elixir's package manager) locally: 

    mix local.hex

    If you already have a version of Hex installed, you'll be asked whether you want to upgrade Hex to the latest version.

  2. Then install the Phoenix 1.3 archive: 

    mix archive.install https://github.com/phoenixframework/archives/raw/master/phx_new.ez

  3. With Phoenix successfully installed, you can now create a Phoenix project which we named refuge: 

    mix phx.new refuge

    You'll see a list of files and directories created by the generator in a new refuge parent directory.

  4. Settle into the project by changing into the refuge directory.

  5. Generate the scaffold HTML app using the phx.gen.html generator: 

    mix phx.gen.html Wildthings Bear bears name:string type:string hibernating:boolean

  6. Then pop open the refuge directory in your favorite editor and you should see all the code we walked through in the video!

Route vs. Match Functions

Our route function currently takes one argument: a map representing the conversation. We then pattern match the key-value pairs in the map. For example, here's the route that matches a GET request for the path /bears: 

def route(%Conv{ method: "GET", path: "/bears" } = conv) do
  ...
end

By comparison, here's the match function generated by Phoenix: 

def match(conn, "GET", ["/bears"]) do
  ...
end

Notice it takes three arguments: conn which is roughly equivalent of our conversation map, the HTTP verb, and a list containing path segments.

Interestingly, you might recall that earlier in the course in the Function Clauses section our route function looked very similar to the match function generated by the Phoenix macros. We had a route/1 function that called a route/3 function, passing the HTTP method and requested path as the second and third arguments respectively: 

def route(conv) do
  route(conv, conv.method, conv.path)
end

And here's the route/3 function that matched a GET request for the path /bears: 

def route(conv, "GET", "/bears") do
  ...
end

That's strikingly similar to the match function generated by Phoenix. One notable difference is match takes as its third argument a list of path segments which can contain more information to pattern match.

Anyway, it's a slightly different design. One of the primary reasons we chose a single map argument was it gave us an interesting opportunity to learn how to pattern match maps. Now that you've learned both ways of doing it, if you prefer the 3-argument design over the 1-argument design then by all means feel free to change the code! After all, experimenting and practicing is an important part of learning.
