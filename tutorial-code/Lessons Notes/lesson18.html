Running Tests

As you saw in the video, you can run a specific test case like so 

mix test test/servy_test.exs

If you want to run multiple test cases, just give mix test a space-separated list of test case file names: 

mix test test/parser_test.exs test/handler_test.exs

And you can run all the test cases in one fell swoop like this 

mix test

Sometimes rather than running all the tests in a test case, it's handy to just run a particular test that's failing. To do that, you can include the line number of the test to run, like so: 

mix test test/parser_test.exs:7

But how do you know the line number? You could get it from your code editor, but there's an easier way! When a test fails, the second line of the output includes the file name and line number where the failing test was defined. For example, the following test failed on line 7: 

1) test parses a list of header fields into a map (ParserTest)
     test/parser_test.exs:7
     Assertion with == failed
     code:  headers == %{"B" => "2"}
     left:  %{"B" => "2", "A" => "1"}
     right: %{"B" => "2"}
     stacktrace:
       test/parser_test.exs:12: (test)

The easiest way to run a failing test is to copy the second line of the output above (the file name and line number) and paste it after mix test, so you end up with: 

mix test test/parser_test.exs:7

The mix test task has a bunch of other options you may want to check out in the documentation or by running: 

mix help test

Exercise: Check Out Assertion Macros

We personally like a unit testing framework that doesn't try to get too fancy. And ExUnit feels like it's not too much, and not too little... but just right!

The vast majority of the time you'll use two assertion macros: assert and refute. Because generally speaking, you expect something to either be truthy or falsey. ExUnit does provides a few other special-case assertion macros and it's worth spending a minute looking through the other assertions just so you know what's possible.

Trick: Speeding Up Tests

By default, ExUnit executes each test case (test file/module) serially. Thus, as the number of test cases increases, so does the time it takes to execute those tests. You can speed up the execution of multiple test cases by running them concurrently rather than serially. To allow a test case to run concurrently, simply set the async option to true on this existing line: 

use ExUnit.Case, async: true

It's important to note that individual tests within the test case are always run serially. Setting async to true allows the test case as a whole to run in parallel with other test cases.

With only a couple test cases as we have now, you likely won't notice a difference. This is just a trick to keep in your back pocket when/if test execution speed becomes an issue. Also, beware that you won't be able to use this trick if a test case accesses shared state or resources with another test cases. If both those test cases run concurrently, then they'll end up stepping on each other.

Tip: Organizing Doctests

In the video we added the doctest macro to the ParserTest test case like so: 

defmodule ParserTest do
  use ExUnit.Case
  doctest Servy.Parser

  ...
end

That line tells ExUnit to look for doctests in the Servy.Parser module and generate ExUnit tests for any doctests it finds. Any generated ExUnit tests will be included in the existing ParserTest test case.

Another option is to define a new test case who's entire job is to aggregate all the doctests. For example, you could create a DocTest module that simply has one or more doctest lines: 

defmodule DocTest do
  use ExUnit.Case
  doctest Servy.Parser
  doctest Servy.Handler
  doctest Servy.Plugins
end

Now any generated ExUnit tests for those doctests will be included in the DocTest test case. So to run all the doctests you would use 

mix test test/doc_test.exs

It's just a different way to organize, and run, doctests.

HTTP Message Spec

In the video we added CRLF characters to the end of the request lines in the tests and to the end of the response lines returned by Handler.format_response. We did this because later on we'll send requests from a browser, and browsers follow the HTTP specification (RFC 2616). So we want our tests to accurately reflect requests that a browser will send, and we want to format responses in a way the browser expects.

If you're curious, the HTTP Message section of RFC 2616 specifies where CRLF characters are required in both requests and responses.
